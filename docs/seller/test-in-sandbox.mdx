---
title: "Test in Sandbox"
description: "Test seller agent payment flows safely before production."
---

Sandbox testing proves that your route can issue 402 requirements, open tabs, verify guarantees, settle, serve paid work, and produce useful logs.

Use Base Sepolia (`eip155:84532`) and small prices while you are validating behavior.

## What to test first

<Steps>
  <Step title="Unpaid request">
    Call the protected route without a payment header. Confirm your API returns HTTP 402 with the expected scheme, price, network, asset, `payTo`, and tab endpoint.
  </Step>
  <Step title="Paid retry">
    Use a 4Mica-capable buyer client to open a tab, sign a guarantee, retry the request, and receive the protected response.
  </Step>
  <Step title="Invalid payment">
    Retry with a missing, expired, malformed, or mismatched payment header. Confirm the paid handler does not run.
  </Step>
  <Step title="Delivery evidence">
    Confirm logs connect request ID, tab ID, payer, route, price, settlement result, and response hash or delivery marker.
  </Step>
  <Step title="Claim path">
    Exercise the default path in a controlled environment so your team understands when a seller can claim from collateral and when a claim expires.
  </Step>
</Steps>

## Sandbox checklist

- Use a test seller wallet and a test buyer wallet.
- Keep the first price low, such as `$0.01`.
- Test one protected route before adding tiers or metered pricing.
- Confirm your tab endpoint is reachable from the buyer client.
- Confirm your server calls the facilitator from server-side code.
- Confirm repeated invalid attempts trigger rate limits or alerts.
- Confirm support can find a paid request from a tab ID or request ID.

## Abuse scenarios to simulate

- A buyer without enough collateral tries to pay.
- A buyer pays for a cheap route and reuses the header on an expensive route.
- A buyer changes request parameters after receiving a quote.
- A buyer sends many unpaid requests to force expensive pre-work.
- A buyer claims the agent failed after receiving a response.
- Another client tries to impersonate a known payer or agent.

The expected result is simple: unpaid or invalid requests do not trigger paid work, and every accepted request has a payment record tied to the delivered result.

## When sandbox is done

Move toward production only after the payment flow is boring. You should be able to answer:

- What did the buyer pay for?
- Which wallet paid?
- Which seller address received the obligation?
- Which route and price were used?
- Did verification and settlement succeed?
- What did the agent return?
- What happens if the payer does not settle the tab later?

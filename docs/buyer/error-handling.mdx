---
title: "Error Handling"
description: "Handle buyer-side payment, collateral, and retry failures."
---

Buyer errors should fail closed. If the agent cannot verify a seller, fit a payment inside policy, sign a guarantee, or connect the payment to the task, it should stop or ask for approval instead of spending blindly.

## Common error categories

| Error | Meaning | Buyer response |
| --- | --- | --- |
| Unsupported network | Seller requires a network your client has not registered. | Skip, choose another seller, or add the network intentionally. |
| Missing collateral | Wallet cannot back the payment guarantee. | Add collateral or lower the task budget. |
| Price over limit | Seller price exceeds policy. | Ask for approval, choose another seller, or stop. |
| Seller mismatch | Domain, tab endpoint, or `payTo` address does not match expectations. | Do not sign. Treat as suspicious. |
| Invalid signature | Payment signing failed or produced an invalid payload. | Retry only after checking signer and network configuration. |
| Seller verification failure | Seller rejected the payment retry. | Log the failure and avoid repeated automatic retries. |
| Task budget exhausted | The agent has spent its allowed budget. | Stop or ask for a higher budget. |

## Retry rules

Do not retry forever. Automated retries can turn a small issue into unexpected spend or noisy traffic.

Use strict retry rules:

- retry network timeouts with backoff;
- do not retry policy failures automatically;
- do not retry payments over budget;
- do not retry mismatched seller addresses;
- cap attempts per seller and per task;
- record the final failure reason in the task log.

## User-facing status

Keep payment state understandable:

- `needs_approval`: the agent found a payment that exceeds policy;
- `insufficient_collateral`: the wallet cannot back the payment;
- `blocked_seller`: policy does not allow this seller;
- `payment_failed`: signing, tab opening, or seller verification failed;
- `budget_exhausted`: the task limit has been reached;
- `paid`: the request was paid and a response was received.

## What to log

Log enough to debug without guessing:

- task ID and request ID;
- seller domain and `payTo` address;
- amount, network, and asset;
- policy version;
- failure reason;
- retry count;
- whether human approval was requested;
- final task outcome.

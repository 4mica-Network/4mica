Use these patterns as a starting point for agents that need to pay services during a task while staying inside budget and policy.

## Basic paid request

```ts
const response = await fetchWithPayment("https://api.example.com/data");

if (!response.ok) {
  throw new Error(`Paid request failed: ${response.status}`);
}

const data = await response.json();
```

## Task budget guard

```ts
type BuyerPolicy = {
  maxPerRequestUsd: number;
  maxPerTaskUsd: number;
  allowedDomains: string[];
};

function canPay(requirement: {
  amountUsd: number;
  sellerDomain: string;
  taskSpendUsd: number;
}, policy: BuyerPolicy) {
  if (!policy.allowedDomains.includes(requirement.sellerDomain)) return false;
  if (requirement.amountUsd > policy.maxPerRequestUsd) return false;
  if (requirement.taskSpendUsd + requirement.amountUsd > policy.maxPerTaskUsd) {
    return false;
  }
  return true;
}
```

## Approval threshold

```ts
async function requireApprovalIfNeeded(amountUsd: number) {
  if (amountUsd <= 1) return true;

  return await requestHumanApproval({
    title: "Approve agent payment",
    amountUsd,
    reason: "The agent needs a paid data source to finish the task.",
  });
}
```

## Payment log shape

```ts
type PaidRequestLog = {
  taskId: string;
  requestId: string;
  sellerDomain: string;
  payTo: `0x${string}`;
  amountUsd: number;
  network: string;
  asset: string;
  tabId: string;
  policyVersion: string;
  reason: string;
  resultStatus: "paid" | "failed" | "blocked" | "needs_approval";
};
```

## Workflow pattern

1. Agent plans the task.
2. Agent estimates which paid services may be needed.
3. Policy sets max spend for the task.
4. Each payment is checked against seller, amount, network, and reason.
5. The task log links paid requests to outputs.
6. The user can audit spend after completion.

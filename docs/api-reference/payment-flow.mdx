# Payment Flow

The 4Mica payment flow keeps HTTP simple while moving settlement complexity into tabs, signatures, facilitator calls, and protocol enforcement.

## Sequence

1. A payer deposits ETH or a supported ERC-20 into the Core4Mica vault.
2. A seller protects an HTTP route with x402 payment requirements and advertises a tab endpoint.
3. The buyer receives HTTP 402, opens or reuses a payment tab, signs a guarantee, and retries the request.
4. The seller verifies and settles the payment through the facilitator, then serves the protected response.
5. The payer settles the tab after the credit window, or the seller claims from collateral during the on-chain claim window if the payer defaults.

## Headers

- V1 payment payloads commonly travel as X-PAYMENT.
- V2 payment payloads commonly travel as PAYMENT-SIGNATURE.
- Both bind claims such as tab ID, request ID, amount, recipient, asset, timestamp, and version.
- The server should verify and settle before serving protected content.

## Settlement Outcomes

- Paid: payer settles the tab and collateral unlocks according to protocol rules.
- Remunerated: seller claims collateral using a valid certificate during the claim window.
- Expired: seller missed the claim window and can no longer claim that collateral for the tab.
- Invalid: verification or validation failed and the resource should not be served as paid.

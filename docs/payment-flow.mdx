---\n\n## Payment Flow Overview\n\nThe x402 Payment Protocol implements HTTP 402 Payment Required, enabling servers to request payment before granting access to resources.\n\n## Step-by-Step Flow\n\n<Steps>\n  <Step title=\"Client Request\">\n    User makes a request to a protected resource:\n\n    ```http\n    GET /api/protected HTTP/1.1\n    Host: api.myservice.com\n    Authorization: Bearer user_token\n    ```\n  </Step>\n\n  <Step title=\"Server Checks Access\">\n    The server checks if the client has valid payment credentials:\n\n    ```typescript\n    app.get('/api/protected', async (req, res) => {\n      const hasValidPayment = await verifyPaymentGuarantee(\n        req.headers['x-payment-guarantee']\n      );\n\n      if (!hasValidPayment) {\n        return res.status(402).json({\n          paymentRequired: {\n            tabUrl: 'https://app.4mica.com/tab/xxx',\n            amount: '0.01',\n            currency: 'ETH',\n          }\n        });\n      }\n      // Grant access\n    });\n    ```\n  </Step>\n\n  <Step title=\"Client Opens Payment Tab\">\n    The client opens the payment tab URL in a modal or separate window:\n\n    ```typescript\n    const response = await fetch('/api/protected');\n    if (response.status === 402) {\n      const { paymentRequired } = await response.json();\n      window.open(paymentRequired.tabUrl, '_blank');\n    }\n    ```\n  </Step>\n\n  <Step title=\"User Funds Tab\">\n    User deposits funds into the payment tab:\n\n    - Connects wallet\n    - Signs transaction\n    - Funds tab (e.g., 0.01 ETH)\n    - Tab balance is now available\n  </Step>\n\n  <Step title=\"Client Signs Guarantee\">\n    Once funded, the client signs a payment guarantee:\n\n    ```typescript\n    const guarantee = await signGuarantee({\n      tabId: paymentResponse.tabId,\n      amount: '0.01',\n      requestHash: hashRequest(originalRequest),\n      schemeAddress: '0x...',\n      validationPolicy: paymentRequired.validationPolicy,\n    });\n    ```\n  </Step>\n\n  <Step title=\"Client Retries with Guarantee\">\n    The client retries the original request with the payment guarantee:\n\n    ```http\n    GET /api/protected HTTP/1.1\n    Host: api.myservice.com\n    Authorization: Bearer user_token\n    X-Payment-Guarantee: {signature}\n    X-Payment-Guarantee-Tab: tab_xxx\n    ```\n  </Step>\n\n  <Step title=\"Server Validates Guarantee\">\n    The server verifies the signature and guarantee:\n\n    ```typescript\n    const isValid = await schemeContract.verifyGuarantee(\n      guarantee.signature,\n      guarantee.tabId,\n      guarantee.amount,\n      guarantee.validationPolicy\n    );\n\n    if (isValid) {\n      // Grant access and record payment\n      res.json({ data: 'Protected content' });\n    }\n    ```\n  </Step>\n\n  <Step title=\"Settlement\">\n    At regular intervals, the facilitator settles accumulated payments:\n\n    ```typescript\n    const guarantees = await getVerifiedGuarantees();\n    const settlement = await settleFacilitatorPayments(guarantees);\n    // On-chain settlement occurs\n    ```\n  </Step>\n</Steps>\n\n## V1 vs V2 Flows\n\n### V1: Standard Payment\n\n```mermaid\nsequenceDiagram\n    Client->>Server: GET /protected\n    Server->>Server: Check payment\n    Server-->>Client: 402 + Payment Requirement\n    Client->>FacilitatorUI: Open payment tab\n    Client->>Client: Sign guarantee\n    Client->>Server: GET /protected + Guarantee\n    Server->>Scheme: Verify signature\n    Scheme-->>Server: ✓ Valid\n    Server-->>Client: 200 + Data\n```\n\n### V2: Verified Payment\n\n```mermaid\nsequenceDiagram\n    Client->>Server: GET /protected\n    Server->>Server: Check payment\n    Server-->>Client: 402 + Payment Requirement\n    Client->>FacilitatorUI: Open payment tab\n    Client->>Client: Sign guarantee\n    Client->>Server: GET /protected + Guarantee\n    Server->>Scheme: Verify signature\n    Scheme-->>Server: ✓ Pending (V2)\n    Server-->>Client: 202 + Job ID\n    Note over Server,Validator: Job execution\n    Validator->>Validator: Complete validation\n    Validator->>OnChain: Record score\n    OnChain-->>Facilitator: Settlement\n    Facilitator-->>Client: Settlement complete\n```\n\n## Message Formats\n\n### 402 Payment Required Response\n\n```json\n{\n  \"paymentRequired\": {\n    \"tabUrl\": \"https://app.4mica.com/tab/tab_xxx?facilitator=fac_xxx\",\n    \"guaranteeRequired\": {\n      \"amount\": \"0.01\",\n      \"currency\": \"ETH\",\n      \"validationPolicy\": {\n        \"address\": \"0x...\",\n        \"method\": \"erc8004\",\n        \"minimumScore\": 80\n      }\n    },\n    \"requestHash\": \"0x...\",\n    \"scheme\": \"0x...\",\n    \"nonce\": 12345\n  }\n}\n```\n\n### Payment Guarantee Header\n\n```\nX-Payment-Guarantee: eyJhbGciOiJFQzI1NksifQ...signature\nX-Payment-Tab-Id: tab_xxx\nX-Payment-Nonce: 12345\nX-Payment-Chain: eip155:8453\n```\n\n## Error States\n\n### Invalid Guarantee\n\n```json\n{\n  \"error\": {\n    \"code\": \"INVALID_GUARANTEE\",\n    \"message\": \"Payment guarantee verification failed\",\n    \"reason\": \"signature_mismatch\"\n  }\n}\n```\n\n### Insufficient Funds\n\n```json\n{\n  \"status\": 402,\n  \"paymentRequired\": {\n    \"amount\": \"0.01\",\n    \"tabBalance\": \"0.005\",\n    \"shortfall\": \"0.005\"\n  }\n}\n```\n\n### Expired Guarantee\n\n```json\n{\n  \"error\": {\n    \"code\": \"GUARANTEE_EXPIRED\",\n    \"message\": \"Payment guarantee has expired\",\n    \"expiryTime\": \"2024-01-01T00:10:00Z\"\n  }\n}\n```\n\n## Network Interactions\n\n### On-Chain Verification\n\nThe scheme contract verifies guarantees on-chain:\n\n```solidity\nfunction verifyGuarantee(\n    bytes signature,\n    bytes32 tabId,\n    uint256 amount,\n    address validationPolicy\n) external view returns (bool) {\n    // Recover signer from signature\n    address signer = recoverSigner(signature);\n    \n    // Check tab has sufficient balance\n    require(tabs[tabId].balance >= amount);\n    \n    // Check signature matches request\n    require(signer == tabs[tabId].owner);\n    \n    return true;\n}\n```\n\n### Settlement Process\n\n1. **Accumulate**: Facilitator collects verified guarantees\n2. **Batch**: Group guarantees for efficiency\n3. **Verify**: Server-side validation of all guarantees\n4. **Settle**: Execute on-chain settlement transaction\n5. **Distribute**: Allocate funds to facilitators and operators\n\n## Guarantee Lifecycle\n\n```\n┌─────────────┐\n│   Created   │ (signed by client)\n└──────┬──────┘\n       │\n       ▼\n┌─────────────────────┐\n│  Signature Verified │ (server validates)\n└──────┬──────────────┘\n       │\n       ▼\n┌──────────────────┐\n│ Tab Funded       │ (client deposits)\n└──────┬───────────┘\n       │\n       ▼\n┌──────────────────┐\n│ Claimed by       │ (payment used)\n│ Facilitator      │\n└──────┬───────────┘\n       │\n       ▼\n┌──────────────────┐\n│ V1: Settled      │ (or)\n│ V2: Pending      │\n│    Validation    │\n└──────┬───────────┘\n       │\n       ▼\n┌──────────────────┐\n│ V2: Validated    │ (if applicable)\n└──────┬───────────┘\n       │\n       ▼\n┌──────────────────┐\n│ Final Settlement │ (on-chain)\n└──────────────────┘\n```\n\n## Best Practices\n\n<Tip>\n  **Request Deduplication**: Use nonces to prevent replay attacks. Include a unique value in each guarantee signing.\n</Tip>\n\n<Warning>\n  **Security**: Always verify signatures server-side. Never trust client-provided claims about payment status.\n</Warning>\n\n<Tip>\n  **Performance**: Batch settlement operations to reduce gas costs.\n</Tip>\n\n<Tip>\n  **Monitoring**: Track guarantee lifecycle with webhooks for debugging and auditing.\n</Tip>\n\n## Next Steps\n\n- [Code examples →](/code-examples)\n- [Facilitator API →](/facilitator-api)\n- [Core API →](/operator-api)\n
